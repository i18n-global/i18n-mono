# 커밋 메시지 가이드라인

## 기본 규칙

- 모든 커밋 메시지는 **한국어 동사형**으로 작성합니다
- 정말 필요시에만 2줄까지 허용합니다
- 첫 줄은 50자 이내로 작성합니다

## 커밋 타입

### feat: 기능 구현
새로운 기능을 추가할 때 사용합니다.

```
feat: 다국어 자동 변환 기능 추가
feat: Google Sheets 연동 기능 구현
```

### fix: 버그 수정
버그를 수정할 때 사용합니다.

```
fix: 번역 키 추출 오류 수정
fix: 서버 컴포넌트 타입 에러 해결
```

### docs: 문서 수정
문서만 수정할 때 사용합니다.

```
docs: README 업데이트
docs: API 문서 보완
```

### style: 코드 스타일
코드 포맷팅, 세미콜론 누락 등 기능 변경 없는 스타일 변경입니다.

```
style: 코드 포맷팅 적용
style: 들여쓰기 정리
```

### refactor: 리팩토링
기능 변경 없이 코드를 개선할 때 사용합니다.

```
refactor: 모듈 구조 개선
refactor: 함수 분리 및 최적화
```

### test: 테스트
테스트 코드 추가 또는 수정할 때 사용합니다.

```
test: 번역 함수 단위 테스트 추가
test: 통합 테스트 케이스 보완
```

### chore: 기타 작업
빌드 설정, 패키지 관리 등 기타 작업입니다.

```
chore: 의존성 업데이트
chore: 빌드 스크립트 수정
```

### perf: 성능 개선
성능을 개선할 때 사용합니다.

```
perf: 번역 함수 최적화
perf: 빌드 속도 개선
```

### ci: CI/CD
CI/CD 설정 변경할 때 사용합니다.

```
ci: GitHub Actions 워크플로우 추가
ci: 배포 파이프라인 개선
```

## 예시

### 좋은 예시 ✅

```
feat: 자동 번역 래핑 기능 구현
fix: 타입 에러 해결
docs: 사용 가이드 추가
refactor: AST 변환 로직 개선
test: 번역 추출 테스트 추가
chore: npm 패키지 버전 업데이트
```

### 나쁜 예시 ❌

```
feat: add translation feature (영어 사용)
feat: 기능 추가 (동사형 아님)
feat: 자동 번역 래핑 기능을 추가하고 타입 안전성을 보장하며 성능을 최적화함 (너무 김)
```

## 2줄 사용 예시

정말 필요한 경우에만 2줄을 사용합니다.

```
feat: 서버 컴포넌트 지원 추가
Next.js App Router 환경에서 서버 컴포넌트 번역 기능 구현
```

```
fix: 타입 에러 해결
TypeScript strict 모드에서 발생하는 타입 불일치 문제 수정
```

## 패키지별 커밋

모노레포 구조이므로 패키지명을 명시할 수 있습니다 (선택사항).

```
feat(core): 타입 안전 번역 키 시스템 추가
fix(tools): 번역 추출 버그 수정
docs(demo): 사용 예시 업데이트
```

## 주의사항

- ❌ 영어 사용 금지
- ❌ 명사형 사용 금지 (예: "기능 추가" → "기능 구현")
- ❌ 과도하게 긴 설명 금지
- ✅ 간결하고 명확한 동사형 사용
- ✅ 무엇을 했는지 명확히 표현
